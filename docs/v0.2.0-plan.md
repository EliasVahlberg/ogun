# Ogun v0.2.0 — Implementation Plan (Revised)

> Date: 2026-03-15
> Based on: deviation report + oku developer answers
> Addresses: D1, D2, D3, D12 (revised per oku feedback)

---

## Overview

v0.1.0 shipped the core single-pass algorithm with Lee BFS routing and
hardcoded potential. v0.2.0 closes the gap with the research doc's novel
signature and exposes the metadata oku needs for erosion and road hierarchy.

Five changes, in priority order:

1. **PathFinder negotiation routing** — with route cost + congestion grid output
2. **Per-pair repulsion config** — kernel configuration, not a trait
3. **Per-node metrics on Layout** — accessibility and congestion
4. **Unplaced node tracking** — report which nodes couldn't be placed
5. **Score breakdown** — expose the four component metrics

---

## 1. PathFinder Negotiation Routing

### What changed from original plan

Oku confirmed negotiated routing is the top priority and additionally wants
two pieces of routing metadata exposed on Layout:
- `route_cost: HashMap<EdgeId, f32>` — total congestion cost of each path
- `congestion_grid: Vec<u32>` — per-cell route overlap count

Oku does NOT want per-cell cost map input for v0.2.0 (binary obstacles suffice).

### Design

Two-phase routing (unchanged from original plan):

**Phase 1 — Congestion-aware initial routing (inside placement loop)**

Replace `lee_route()` (BFS, uniform cost) with `negotiate_route()` (Dijkstra,
congestion-weighted). Maintain alongside `blocked`:

```
present_sharing: Vec<u32>  — routes currently using each cell
history: Vec<f32>          — accumulated historical congestion
```

Cost per cell (McMurchie & Ebeling 1995):
```
cost(cell) = (1.0 + history[cell]) * (1 + present_sharing[cell])
```

**Phase 2 — Rip-up and reroute (after all nodes placed)**

Iterate over all edges (highest-weight first):
1. Remove edge from `present_sharing`, reroute with current costs, re-add
2. Update `history` for cells where `present_sharing > 1`
3. Terminate when no shared cells or `max_iterations` reached

**Output metadata:**
- `route_cost[edge_id]` = sum of `cost(cell)` along the final path
- `congestion_grid` = final `present_sharing` vec

### Files

- `src/routing.rs` — add `CongestionState`, `negotiate_route()` (Dijkstra),
  `rip_up_and_reroute()`
- `src/generate.rs` — replace `lee_route()` with `negotiate_route()`, add
  Phase 2 after placement loop, populate new Layout fields
- `src/graph.rs` — add `negotiation_iterations: u32` and
  `history_increment: f32` to `OgunConfig` (defaults: 3, 1.0)
- `src/layout.rs` — add `route_cost` and `congestion_grid` fields

### Breaking changes

- `OgunConfig` gains two fields (provide defaults)
- `Layout` gains two fields
- Route paths will differ — regenerate fixtures

## 2. Per-Pair Repulsion Config

### What changed from original plan

Oku explicitly rejected a `PotentialFn` trait — it would force them to
reimplement ogun internals (distance calc, grid iteration). They want
declarative kernel configuration instead.

Specifically: a per-pair repulsion multiplier so oku can say "sacred↔military
repels 3× harder than the default." Gap (radius inflation) stays oku-side.

### Design

Add to `OgunConfig`:

```rust
/// Per-node-pair repulsion multipliers. If `(a, b)` maps to `3.0`,
/// the repulsion force between nodes a and b is `repulsion_k * 3.0 / dist_sq`.
/// Pairs not in the map use the default multiplier of `1.0`.
/// Symmetric: `(a, b)` and `(b, a)` are equivalent.
pub repulsion_pairs: HashMap<(NodeId, NodeId), f32>,
```

In `utility()`, when computing repulsion for placed node `j` against candidate
position for node `i`, look up `repulsion_pairs.get((i, j))` to get the
multiplier. Default to 1.0 if absent.

### Performance concern

HashMap lookup in the hot loop. Mitigation: convert to a flat lookup structure
in `generate()` before the main loop. For n nodes, a `Vec<Vec<f32>>` indexed
by `[node_i][placed_index]` would be O(1) but requires mapping placed SoA
index back to NodeId. Alternative: `Vec<f32>` of size n×n with
`repulsion_pairs[i * n + j]`. For 500 nodes that's 250K f32s = 1MB, acceptable.

### Files

- `src/graph.rs` — add `repulsion_pairs` to `OgunConfig`
- `src/generate.rs` — pre-build flat repulsion matrix before main loop
- `src/potential.rs` — `utility()` accepts repulsion slice, applies per-pair
  multiplier in the repulsion loop

### Breaking changes

- `OgunConfig` gains a field (default to empty HashMap)

## 3. Per-Node Metrics on Layout

### What changed from original plan

Oku wants per-node metrics, not just global ScoreBreakdown. Specifically:

- `node_accessibility: HashMap<NodeId, f32>` — fraction of incident edges
  that were successfully routed (3/5 routed = 0.6)
- `node_congestion: HashMap<NodeId, f32>` — average congestion cost across
  this node's routed paths

These directly feed erosion durability: low accessibility = vulnerable,
high congestion = important hub.

### Design

Compute after routing is complete:

```rust
// Accessibility: for each node, count routed edges / total edges
for node in &graph.nodes {
    let total = adj[node.id].len();
    let routed = adj[node.id].iter()
        .filter(|(eid, _, _)| paths.contains_key(eid))
        .count();
    node_accessibility[node.id] = routed as f32 / total.max(1) as f32;
}

// Congestion: for each node, average route_cost of its routed edges
for node in &graph.nodes {
    let costs: Vec<f32> = adj[node.id].iter()
        .filter_map(|(eid, _, _)| route_cost.get(eid))
        .copied()
        .collect();
    node_congestion[node.id] = if costs.is_empty() { 0.0 }
        else { costs.iter().sum::<f32>() / costs.len() as f32 };
}
```

### Files

- `src/layout.rs` — add `node_accessibility` and `node_congestion` fields
- `src/generate.rs` — compute after routing, populate Layout

### Breaking changes

- `Layout` gains two fields

## 5. Score Breakdown

### Design (unchanged from original plan)

```rust
pub struct ScoreBreakdown {
    pub path_efficiency: f32,
    pub accessibility: f32,
    pub congestion: f32,
    pub void_ratio: f32,
    pub composite: f32,
}
```

Lower priority than per-node metrics but cheap to add since `scoring.rs`
already computes all four.

### Files

- `src/layout.rs` — `score: ScoreBreakdown`

## Implementation Order

```
1. Unplaced tracking           (D12)  ~15 min     [no fixture change]
2. Score breakdown             (D3)   ~30 min     [fixture format change]
3. Per-pair repulsion config   (D2)   ~1 session  [output change]
4. PathFinder routing + meta   (D1)   ~2 sessions [output change]
5. Per-node metrics            (D3+)  ~30 min     [depends on routing meta]
6. Regenerate fixtures, full test suite
7. Update CHANGELOG, bump to v0.2.0
8. Tag, release, publish
```

Steps 1–2 are independent and non-disruptive — ship them first.
Step 3 changes algorithm output.
Step 4 is the big one — changes routing fundamentally.
Step 5 depends on step 4 (needs route_cost for node_congestion).

Estimated total: 3–4 sessions.

## Out of Scope for v0.2.0

Per oku developer's answers:
- Boundary affinity in potential (v0.3.0)
- Per-cell routing cost map input (v0.3.0)
- Non-rectangular obstacles (v0.3.0)
- Weighted terrain costs (v0.3.0, Saltglass integration)
- PotentialFn trait (rejected — kernel config instead)
- Per-cell route attribution (wait for negotiated routing impact)
- ConnectionDemand wiring (oku-side, not ogun)
- Hierarchical generation support (oku-side orchestration)
